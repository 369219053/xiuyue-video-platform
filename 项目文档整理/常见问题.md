# 常见问题与已修复 Bug 记录

> 遇到问题先看本文档；若涉及取数逻辑，配合 [代码清洗规则](./代码清洗规则.md) 第十四节"检查清单"排查。

## 一、数据取数类

### Q1. 任务取到了错误日期的台词/人物（如要 5.23 却用了 5.16 的）

**症状**：5.23 张海玲组 redhalo关节康 模式 1，发给 Coze 工作流的 `input` 里全是 5.16 的内容。

**根因**：钉钉文档把**同一产品分多段写**（旧段 5.16 + 新段 5.23 在同一 Tab 内分两个独立标题段落），旧版 `scan_dingtalk_products` 把它们当两个独立产品返回，前端拿到的第一段只有 5.16，找不到 5.23 → 模式 1 回退到最近的 5.16。

**修复版本**：V2.3（2026-05-26）

**修复方式**：扫描阶段按 `(sheet, _norm_product(product_name))` 合并所有同名段落的 `templates` 与 `available_dates`，前端卡片只显示一张，取数命中正确日期。

**人工排查**：
1. 看后端日志是否打印过 `ℹ️ 找到 N 个同名段落，合并所有模板行`
2. 看产品卡片 `原料模板:` 行的可用日期是否包含任务日期
3. 不确定时按 [代码清洗规则.md 第十四节] 7 步排查

---

### Q2. UI 同一个产品出现两张卡片，M1 套数不一致

**症状**：同一组同一产品出现两张卡，一张写 `M1: 15套 原料模板 5.16（16 行）`,另一张写 `M1: 7套 原料模板 5.23（10 行）`,点"台词填入"填的是 7。

**根因**：与 Q1 同源——`scan_dingtalk_products` 没合并同名段落。两张卡分别基于各自段落的 `templates` 算 `m1_lyrics`，所以数字对不上。

**修复版本**：V2.3

**修复后**：合并为 1 张卡，`原料模板: 5.16 / 5.23（26 行）`,M1 套数与"台词填入"完全一致。

---

### Q3. 模式 1 取数为什么有时随机有时不随机？

**规则**（参见 [代码清洗规则.md 第八节]）：
- `len(pool_1) >= count` → **直接取前 N 条**，不打乱不重组，最可控
- `len(pool_1) < count` → 走 `_random_combine`，按视觉组/台词组锁定后随机抽，凑足条数

**例**：5.23 那天模板池 10 行，任务要 5 条 → 直接取前 5 条；任务要 12 条 → 走随机组合补齐。

---

### Q4. 模式 2 的"50/50"具体怎么分？

```
n_new = count // 2     # 新日期占一半（向下取整）
n_old = count - n_new  # 旧日期占剩余
```

| 任务条数 | 新日期 | 旧日期 |
|---------|-------|-------|
| 10 | 5 | 5 |
| 11 | 5 | 6 |
| 3 | 1 | 2 |

只找到 1 个日期时退化为"全部用那 1 天"+ ⚠️ 警告。

## 二、服务/部署类

### Q5. `http://118.196.114.163:5678/` 公网访问超时

**先确认服务器内部是否正常**：
```bash
ssh root@118.196.114.163 "pm2 status && curl -s -o /dev/null -w '%{http_code}\n' http://localhost:5678/"
```
返回 `online` + `200` 说明服务正常。

**问题在云端安全组**：
1. 火山引擎控制台 → ECS-Down 实例 → 绑定的安全组
2. 「入方向规则」→「添加规则」→ 协议 TCP、端口 `5678`、来源 `0.0.0.0/0`
3. 保存后 1-2 分钟生效

---

### Q6. PM2 上的 `xiuyue` 进程频繁重启

**排查**：
```bash
ssh root@118.196.114.163 "pm2 logs xiuyue --lines 50 --nostream"
```

常见原因：
- Python 语法错（部署前本地 `python3 -c "import ast; ast.parse(open('xxx.py').read())"` 验证）
- 端口被占用（`ss -tlnp | grep 5678` 看是不是有别的进程占了）
- 依赖缺失（看 traceback 里的 `ModuleNotFoundError`）

---

### Q7. 不要随便 kill Node.js 进程

ECS 上同机部署了多个 Node.js 服务（`qingyun-backend`、`volc-proxy` 等），**严禁使用** `pkill node` / `taskkill /f /im node.exe`。

只用 PM2 操作：
```bash
pm2 restart xiuyue      # 重启
pm2 stop xiuyue         # 停止
pm2 logs xiuyue         # 看日志
```

## 三、钉钉文档约定

### Q8. 钉钉副表里产品段落写多段会有问题吗？

**不会**。系统会自动按"规范化产品名"合并所有同名段落（去括号 + 小写 + 去空格）。

但要遵守：
- **同一产品的多段必须在同一个 Tab（sheet）内** —— 跨 Tab 不合并
- **段落表头第一列必须是"日期"** —— 否则按日期取数会退化为全部模板
- **括号内容会被去除**：`Redhalo关节康（中医版）` 和 `Redhalo关节康` 视为同一产品

---

### Q9. 汇总表「需求数量」列和 UI 上"任务条数"是什么关系？

汇总表里 `(组, 品类, 日期) → 需求数量` 会被 `parse_demand_map` 提取并返回给前端。

点产品卡的 **`✨ 自动填入`** 按钮会把这个值填到对应任务行的"条数"输入框；点 `🗑 清除` 清空。

最终发给 Coze 的实际条数 = UI 上"条数"输入框的值（不是需求数量原值），方便临时调整。
